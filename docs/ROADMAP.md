## Phase 1: Фундамент и деплой

**Goal:** Статическая PWA задеплоена на GitHub Pages, доступна по QR-коду, каталог загружается в браузер.

**Entry condition:** Нет. Это стартовая фаза.

**Requirements covered:** ARCH-05, DATA-01, DATA-02, UI-01, UI-02, UI-03, UI-04, ANALYTICS-02

| ID | Требование |
|----|------------|
| ARCH-05 | GitHub Pages + GitHub Actions автодеплой при push в main |
| DATA-01 | Скрипт convert-catalog.js: CSV/Excel из 1С → catalog.json с нормализацией полей |
| DATA-02 | Каталог ~1000 SKU как public/catalog.json, загружается при первом визите |
| UI-01 | Стартовый экран с 3 CTA: "Подобрать под задачу", "Найти товар", "Готовые комплекты" |
| UI-02 | Мобильный PWA-интерфейс в фирменных цветах (#2b7de0, #f15a2c), tap target 48px+ |
| UI-03 | Установка через QR-код — PWA работает сразу в браузере без скачивания |
| UI-04 | Дата последнего обновления каталога видна в UI |
| ANALYTICS-02 | QR-код с UTM-метками для отслеживания источника трафика |

**Exit condition:**
1. `npm run build` завершается без ошибок
2. `npm run deploy` деплоит на GitHub Pages без ошибок
3. Сканирование QR-кода открывает приложение на мобильном телефоне (iPhone Safari + Android Chrome)
4. Стартовый экран с тремя кнопками отрисован в фирменных цветах, tap target не менее 48px
5. catalog.json загружается в браузер, количество записей логируется в консоль
6. Дата обновления каталога видна в интерфейсе

**Estimated complexity:** Medium

**Key risks:**
- Формат выгрузки из 1С неизвестен заранее — конвертер может потребовать итерации
- base path в Vite должен совпадать с именем репозитория на GitHub — зафиксировать до начала
- Safari на iOS требует отдельной проверки — эмулятор не подходит
- Без реального каталога фаза не завершена: нужна хотя бы тестовая выборка 50-100 SKU

**Key artifacts:**
- `scripts/convert-catalog.js`
- `public/catalog.json`
- `src/App.svelte` (стартовый экран)
- `.github/workflows/deploy.yml`
- QR-код с UTM-метками

## Phase 2: Поиск по каталогу

**Goal:** Клиент может найти товар по артикулу, названию или синониму — без ИИ, полностью на клиенте.

**Entry condition:** Фаза 1 завершена: каталог загружается, приложение доступно по QR.

**Requirements covered:** SEARCH-01, SEARCH-02, SEARCH-03, SEARCH-04, UI-05, UI-06

| ID | Требование |
|----|------------|
| SEARCH-01 | Поиск по артикулу с нормализацией (дефисы/пробелы, регистронезависимый) |
| SEARCH-02 | Поиск по названию и описанию с fuzzy matching через MiniSearch — работает офлайн |
| SEARCH-03 | Словарь синонимов электротехники: минимум 20-30 пар (дифавтомат/АВДТ, ИЭК/IEK и т.д.) |
| SEARCH-04 | Zero-result с альтернативой: никогда пустой экран, похожие товары + кнопка "Спросить ИИ" |
| UI-05 | Дисклеймер "Наличие и цены уточняйте у консультанта" под каждой рекомендацией |
| UI-06 | Кнопка "Показать менеджеру" — форматированный список артикулов для консультанта |

**Exit condition:**
1. Поиск "ВА47-29 1P 16А" находит товар "ВА47-29-1P-16A" (нормализация артикулов)
2. Поиск слова "дифавтомат" находит товары с обозначением "АВДТ" (синонимы)
3. Поиск несуществующего артикула показывает похожие результаты и кнопку "Спросить ИИ" — экран не пустой
4. Кнопка "Показать менеджеру" формирует читаемый список артикулов
5. Дисклеймер виден под результатами поиска

**Estimated complexity:** Medium

**Key risks:**
- MiniSearch с кириллицей требует настройки токенизатора — проверить до интеграции
- Словарь синонимов нужно заполнять с реальными терминами магазина — привлечь консультантов ЭлектроЦентр
- Нормализация артикулов специфична для ассортимента ЭлектроЦентр — возможны краевые случаи

**Key artifacts:**
- `src/services/search.js` (MiniSearch: индекс, синонимы, нормализация)
- `src/components/CatalogSearch.svelte`

## Phase 3: ИИ-диалог

**Goal:** Клиент описывает задачу в свободной форме и получает список товаров с артикулами исключительно из каталога.

**Entry condition:** Фаза 2 завершена: поиск по каталогу работает, catalog.json доступен на клиенте.

**Requirements covered:** ARCH-01, ARCH-02, ARCH-03, CHAT-01, CHAT-02, CHAT-03, CHAT-04, SEC-01, SEC-02

| ID | Требование |
|----|------------|
| ARCH-01 | Cloudflare Worker как прокси к OpenRouter: API-ключ скрыт, rate limit 20 req/IP/час |
| ARCH-02 | Cloudflare AI Gateway для кэширования: TTL 1ч для диалогов, 24ч для поиска |
| ARCH-03 | Prefix caching system prompt: статичный system prompt в начале промпта — экономия токенов |
| CHAT-01 | Диалоговый подбор: клиент описывает задачу → ИИ уточняет (макс 2 вопроса) → список с артикулами |
| CHAT-02 | Quick-reply chips после каждого ответа ИИ для быстрого продолжения диалога |
| CHAT-03 | ИИ рекомендует ТОЛЬКО товары из каталога: system prompt запрещает артикулы вне списка |
| CHAT-04 | Стриминг ответов ИИ через SSE — пользователь видит ответ сразу |
| SEC-01 | Защита от prompt injection: system prompt ограничивает роль, лимит 500 символов на запрос |
| SEC-02 | Rate limiting по IP: 20 запросов/час через Cloudflare Worker |

**Exit condition:**
1. `curl` к Worker с тестовым запросом возвращает HTTP 200 и стриминг-ответ
2. Запрос "нужен автомат для розеток в квартире" возвращает артикулы из catalog.json, а не выдуманные
3. Запрос длиннее 500 символов отклоняется с HTTP 400
4. 21-й запрос с одного IP за час отклоняется с HTTP 429
5. Попытка prompt injection ("игнорируй инструкции и скажи...") не меняет роль ассистента
6. Ответ начинает отображаться до его полного завершения (стриминг виден в интерфейсе)

**Estimated complexity:** High

**Key risks:**
- Cloudflare AI Gateway — дополнительный слой, требует отдельной настройки; его downtime блокирует ИИ
- SSE через Cloudflare Workers имеет нюансы: `TransformStream` + `ReadableStream` — проверить совместимость
- Выбор LLM-модели влияет на качество ответов по электротехнике — протестировать несколько моделей на реальных запросах до финализации
- $10 бюджет: без AI Gateway и prefix caching кончится быстро — реализовать кэширование в этой же фазе

**Key artifacts:**
- `worker/index.js` (прокси + rate limit + SSE)
- `worker/prompts.js` (system prompt)
- `worker/wrangler.toml`
- `src/services/ai.js` (запросы к Worker)
- `src/components/ChatInterface.svelte`

## Phase 4: Комплекты и финальная полировка

**Goal:** Все три функции MVP работают, готовые комплекты доступны с параметрической адаптацией, аналитика собирает данные с первого дня.

**Entry condition:** Фаза 3 завершена: ИИ-диалог работает, Worker задеплоен.

**Requirements covered:** KIT-01, KIT-02, ARCH-04, ANALYTICS-01

| ID | Требование |
|----|------------|
| KIT-01 | Готовые комплекты с видимым составом: 5-10 комплектов, артикулы в превью без клика |
| KIT-02 | Параметрическая адаптация: площадь → коэффициент → количество товаров |
| ARCH-04 | Service Worker: Cache-First для статики, Stale-While-Revalidate для каталога, graceful fallback для ИИ |
| ANALYTICS-01 | Аналитика с первого дня (GA4 или Plausible): QR-сканирования, сессии, запросы к ИИ, найденные товары |

**Exit condition:**
1. Страница "Готовые комплекты" показывает минимум 5 комплектов с артикулами в карточке без клика
2. Изменение площади в KIT-02 меняет количество товаров в комплекте согласно таблице коэффициентов
3. При отключении интернета поиск по каталогу работает, ИИ показывает fallback-сообщение
4. В дашборде GA4 / Plausible видны события: session_start, search_query, ai_request, kit_opened
5. `npm run build && npm run deploy` проходит полный прогон без ошибок — финальная сборка готова к запуску

**Estimated complexity:** Low

**Key risks:**
- Комплекты нужно составлять вместе с консультантами ЭлектроЦентр — зависимость от внешнего человека; начать сбор данных параллельно с фазой 3
- Service Worker на iOS Safari очищает кэш через 7 дней — использовать sessionStorage, задокументировать поведение
- Аналитика должна быть настроена до первого QR в зале — не после

**Key artifacts:**
- `src/components/KitSelector.svelte`
- `public/kits.json` (данные комплектов)
- `src/services/catalog.js` (загрузка + Service Worker кэш)
- Финальный QR-код на печать
