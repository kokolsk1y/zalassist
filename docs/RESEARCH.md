# ZalAssist — Синтез исследований

**Проект:** ZalAssist — PWA ИИ-помощник для клиентов ЭлектроЦентр  
**Дата:** 2026-04-01  
**Источники:** stack.md, features.md, architecture.md, pitfalls.md

---

## Executive Summary

- **Стек подтверждён всеми 4 исследованиями:** Svelte 5 + Cloudflare Workers + MiniSearch + OpenRouter через прокси. Альтернативы хуже по всем критериям — отклонены.
- **Архитектура двухуровневая и бессерверная:** статика на GitHub Pages + edge-прокси на Workers. Для $10 бюджета это единственная рабочая схема.
- **ИИ не работает с каталогом напрямую** — нужен client-side pre-filter: MiniSearch сужает до 20-30 позиций → передаёт в промпт. Без этого галлюцинации неизбежны.
- **Главный риск проекта — не технический:** QR-воронка, которую никто не сканирует, и каталог, который никто не обновляет. Оба убивают проект независимо от качества кода.
- **B2B (прорабы) — основной путь:** быстрый поиск по артикулу и комплектам. Диалог с ИИ — вторичный путь для B2C. Интерфейс должен отражать эту иерархию.
- **Три фичи MVP имеют готовые паттерны** (Lowe's Mylow, Baymard Institute) — изобретать не нужно. Важно: максимум 2 уточняющих вопроса до первого результата.
- **$10 бюджета хватит на ~12 500 диалогов** при правильном кэшировании (Cloudflare AI Gateway + prefix caching). Без кэширования — только ~3 100.
- **PWA на iOS — отдельная точка риска:** Safari ведёт себя иначе, тест на реальном iPhone обязателен до запуска.

## Архитектурные решения

### Двухуровневая статическая архитектура

```
[Телефон клиента]
       | QR-код
       v
[GitHub Pages: статическая PWA]
  |- index.html + JS bundle (~6 KB)
  |- catalog.json (каталог из 1С)
  |- manifest.json + sw.js
       | fetch() к Worker
       v
[Cloudflare Workers: edge proxy]
  |- Добавляет system prompt + API key
  |- Cloudflare AI Gateway (кэш + мониторинг)
  |- Rate limiting (20 req/IP/час)
       |
       v
[OpenRouter → LLM]
```

**Почему не проще:** API-ключ в фронтенде — критическая уязвимость. Worker обязателен.

### Хранение каталога

Каталог хранится как `public/catalog.json` на GitHub Pages. Обновление — git push нового файла. Для v1 это достаточно.

- 10 000 SKU × 300 байт = ~3 MB — допустимо для мобильного
- Service Worker кэширует файл стратегией Stale-While-Revalidate
- В UI показывать дату обновления каталога — обязательно

**Путь к росту:** Workers KV (Фаза 2) → D1 + webhook из 1С (Фаза 3). Архитектура это позволяет без переписывания.

### Контекстная стратегия для ИИ

Выбрана стратегия **Client-Side Pre-filter**:

1. Пользователь вводит запрос
2. MiniSearch на клиенте → топ 20-30 товаров
3. Worker собирает промпт: system prompt + 30 товаров + 3-5 последних сообщений + запрос
4. Итого: ~3 200 токенов на запрос, стоимость ~$0.001

Альтернатива "полный каталог в промпт" (~$0.01-0.05/запрос) отклонена — дорого и хуже качеством из-за эффекта "lost-in-middle". RAG с векторной базой отклонён — избыточная инфраструктура для v1.

### Кэширование (3 уровня)

| Уровень | Механизм | TTL | Эффект |
|---------|---------|-----|--------|
| Браузер | sessionStorage | сессия | Повторный поиск не идёт к API |
| Edge | Cloudflare AI Gateway | 1ч (диалог) / 24ч (поиск) | Одинаковые запросы не тарифицируются |
| Промпт | OpenRouter Prefix Caching | авто | System prompt = 10% цены при повторах |

Результат: $10 → ~12 500 диалогов вместо ~3 100.

### Офлайн-стратегия

| Ресурс | Стратегия SW | Офлайн-поведение |
|--------|-------------|------------------|
| HTML/JS/CSS | Cache-First | Работает |
| catalog.json | Stale-While-Revalidate | Работает (устаревшая версия) |
| ИИ-запросы | Network-Only | Graceful fallback: "ИИ временно недоступен. Поиск по каталогу работает." |

## Паттерны фич

### Фича 1: Подбор товаров через диалог с ИИ

**Паттерн:** Project-first dialog (доказан Lowe's Mylow — конверсия в 2x выше поиска по спецификациям)

**Структура взаимодействия:**
1. Стартовый экран: 3 CTA-кнопки — "Подобрать под задачу / Найти по артикулу / Готовые комплекты"
2. Пользователь описывает задачу свободным текстом
3. ИИ задаёт не более 2 уточняющих вопросов (площадь / тип помещения / мощность)
4. ИИ выдаёт список товаров с артикулами из каталога
5. Quick-reply chips после каждого ответа — для B2B-пользователей, которым нужна скорость

**Почему не wizard:** Цепочка фиксированных вопросов ломается при нестандартных задачах. Свободный диалог гибче — при ограничении в 2 уточнения он не затягивается.

**RAG в промпт обязателен:** до передачи запроса к Worker — прогнать через MiniSearch, топ-20 результатов добавить в system prompt. Без этого ИИ галлюцинирует артикулы.

### Фича 2: Поиск по артикулу, названию, описанию

**Паттерн:** Гибридный поиск — fuzzy matching + нормализация + синонимы

**Нормализация артикулов (реализовать до запуска):**
- Убрать дефисы, пробелы, привести к нижнему регистру
- "ВА47-29 1P 16А" должен находить "ВА47-29-1P-16A"

**Обязательный словарь синонимов:**
- дифавтомат / АВДТ / дифференциальный автомат
- автомат / АВ / автоматический выключатель
- УЗО / устройство защитного отключения
- щиток / щит / распределительный щит
- ИЭК / IEK / EKF / ЭКФ / Schneider / Шнайдер

**Zero-result — не тупик:** при отсутствии результатов всегда показывать похожие + кнопку "Спросить ИИ".

### Фича 3: Готовые комплекты

**Паттерн:** Baymard Institute — содержимое комплекта видно в карточке без клика внутрь.

**Формат карточки:**
- Название ("Комплект для квартиры 50 кв.м.")
- Описание задачи ("Защита 3-комнатной квартиры: освещение + розетки + UЗО")
- 3-4 позиции с артикулами — прямо в превью
- Кнопка "Полный список"
- Кнопка "Показать менеджеру" (открывает список для передачи)

**Параметрическая адаптация:** масштабирование по площади через таблицу коэффициентов — без ИИ. Пример: 50 кв.м. × коэффициент = количество автоматов. Достаточно для v1.

## Риски и митигации

### Критические (убивают проект)

**1. ИИ галлюцинирует товары**
Что: LLM выдаёт несуществующие артикулы или характеристики. Прецеденты: Chevrolet, Air Canada.  
Митигация: system prompt ограничен списком из каталога; артикулы берутся из JSON, не генерируются; дисклеймер "Наличие уточняйте у консультанта".

**2. QR-воронка не работает**
Что: никто не сканирует — 53% уходят при загрузке >3 сек, QR без контекста игнорируется.  
Митигация: текст рядом с QR — конкретный ("Найди автомат за 30 сек"), не абстрактный ("Помощник"); QR минимум A5, на уровне глаз; первый экран < 2 сек на 4G.

**3. Устаревший каталог**
Что: клиент приходит с артикулом, которого нет — доверие сломано навсегда.  
Митигация: договориться о регулярном обновлении ДО запуска; показывать дату обновления в UI; не показывать цены в MVP — только наличие и описание.

### Высокие (создают серьёзные проблемы)

**4. Prompt Injection (OWASP LLM Top 1, 2025)**
Что: клиент заставляет бота сказать что-то неуместное, скриншот попадает в сеть.  
Митигация: роль жёстко ограничена электротехникой ЭлектроЦентр в system prompt; лимит запроса 500 символов; rate limit 20 req/IP/час.

**5. OpenRouter как единая точка отказа**
Что: downtime, смена цен, снятие модели.  
Митигация: fallback — поиск и комплекты работают без ИИ; модель вынесена в переменную окружения; зарезервировать $3 из $10 на демо руководству.

**6. Scope creep при solo-разработке**
Что: "а давайте добавим личный кабинет / историю / рейтинги" — MVP не выходит никогда.  
Митигация: 3 функции зафиксированы письменно; рабочий прототип через 1 неделю; любые дополнения — только после согласования.

**7. PWA на iOS / Safari**
Что: Safari очищает localStorage через 7 дней, ограничивает API Push, ведёт себя иначе чем Chrome.  
Митигация: использовать sessionStorage вместо localStorage; тест на реальном iPhone до запуска — не эмулятор.

**8. Нет метрик — нет доказательств**
Что: без аналитики невозможно доказать 15%-й порог руководству и продолжить проект.  
Митигация: GA4 или Plausible установить до первого запуска; QR с UTM-метками; минимальный дашборд: сканирований / открытых сессий / запросов к ИИ.

## Конфликты между исследованиями

### CONFLICT-01: B2B vs B2C приоритет — RESOLVED

**Конфликт:** PROJECT.md называет B2B (прорабы, закупщики) основным сегментом. features.md отмечает, что все 3 фичи MVP по дизайну B2C-ориентированы (диалог, объяснения, комплекты).

**Разрешение:** Противоречие устраняется иерархией интерфейса, а не выбором одного сегмента. B2B — primary path: быстрый поиск по артикулу, вход в комплект напрямую, quick-reply chips вместо длинного диалога. B2C — secondary path: диалог с объяснениями. Оба пути реализуются через единый интерфейс — разница в том, куда пользователь нажимает первым.

### CONFLICT-02: Хостинг GitHub Pages vs Cloudflare Pages — RESOLVED

**Конфликт:** stack.md рекомендует GitHub Pages как хостинг. architecture.md упоминает миграцию на Cloudflare Pages как следующий шаг роста (Фаза 1).

**Разрешение:** GitHub Pages для v1 — правильный выбор. Нулевая настройка, бесплатно, HTTPS. Cloudflare Pages — опция при росте нагрузки, не требование с первого дня. Противоречия нет — это дорожная карта, а не выбор.

## Открытые вопросы

1. **Имя репозитория на GitHub** — определяет base path в Vite и финальный URL QR-кода. Нужно зафиксировать до начала разработки.

2. **Объём каталога (SKU)** — если >50 000, клиентский MiniSearch не справится, нужен серверный поиск. Текущая архитектура рассчитана на ≤10 000.

3. **Формат выгрузки из 1С** — CSV или JSON, какие поля есть (артикул, наименование, описание, наличие, категория). От этого зависит скрипт `convert-catalog.js`.

4. **Ответственный за обновление каталога** — кто и с какой периодичностью делает выгрузку и push. Без договорённости до запуска — PITFALL-02 неизбежен.

5. **Начальный набор комплектов** — кто составляет, сколько штук на старте. Без готовых комплектов третья фича не работает в день запуска.

6. **Выбор LLM-модели** — OpenRouter предлагает несколько вариантов с разным соотношением цена/качество. Нужно протестировать на реальных запросах электротехники до фиксации в переменной окружения.
